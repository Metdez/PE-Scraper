Implement PIPE-01 (adaptive page selection with skip-lists and 403 fallback) and PIPE-02 (manual
HTML decongestion) exactly as RESEARCH.md's live-verified corrections require: `AdaptiveCrawler`

one plan keeps that dependency (crawl calls decongest per page) inside a single reviewable unit.

offline against real local HTML strings (no network needed), crawl.py offline against mocked

shape" pattern.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Arbitrary firm website -> crawl.select_pages | The crawled site is fully untrusted third-party content; crawl.py only ever extracts URLs/HTML for later decongestion, it does not execute anything from the page |
| decongest.decongest -> HTML parsing | `cleaned_html` originates from an untrusted site; the markdown generator is a well-tested crawl4ai library component, not custom HTML/JS parsing |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-04 | Denial of Service | crawl.select_pages | medium | mitigate | `page_timeout=15000` set explicitly on every fallback `arun()` call; `AdaptiveConfig.max_pages=5` bounds the adaptive crawl itself; a single hanging/slow page cannot make `run-firm` hang indefinitely because crawl4ai's own per-request timeout applies to every fetch, not just the fallback ones |
| T-02-05 | Tampering (SSRF-adjacent) | crawl.select_pages well-known-path fallback | low | accept | Fallback paths are fixed literals (`/about`, `/investment-criteria`, `/strategy`, `/approach`) concatenated onto the firm's own already-supplied base URL — never a user- or page-controlled arbitrary host/path, so there is no attacker-reachable redirect surface introduced here |
</threat_model>
