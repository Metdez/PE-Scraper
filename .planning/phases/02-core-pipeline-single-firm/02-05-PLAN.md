<task type="auto" tdd="true">
  <name>Task 2: ingest_csv — orchestrated seed-and-merge into pipeline.db</name>
  <files>src/pescraper/ingest.py, tests/test_ingest.py</files>
  <read_first>
    src/pescraper/merge.py (`merge_firm_record`, `LIFECYCLE_FIELDS`) — this task's central
    dependency. CONTEXT.md: "Merge-rule scope is universal... a new value only overwrites a
    field if it is non-null; null never clears a previously confirmed value" and "only flag as
    conflicting when ranges don't overlap at all."
  </read_first>
  <behavior>
    - ingest_csv over a 2-row CSV with a brand-new website for each row inserts two new firms rows, each with status left at the FirmRecord default (pending) and ranges parsed from the "EBITDA Range" column.
    - ingest_csv over a CSV row whose website already exists in the DB with status="complete" leaves that firm's status as "complete" after ingest (merge.merge_firm_record's LIFECYCLE_FIELDS pass-through) — it is NOT reset to pending.
    - ingest_csv over a CSV row whose website already has a confirmed, non-overlapping ebitda range sets that firm's needs_review=True after ingest (merge_firm_record reports an "ebidta" conflict; ingest_csv explicitly overrides needs_review=True for a conflicted firm, bypassing the lifecycle pass-through for that one case).
    - A CSV row missing both firm_name and website is skipped (logged, not raised) and does not appear in the resulting firms table; the rest of the file still ingests.
  </behavior>
  <action>
    Define a small `IngestSummary` dataclass (`rows_read: int`, `rows_seeded: int`,
    `rows_skipped: int`, `rows_conflicted: int`). Implement `ingest_csv(csv_path: str | Path,
    conn: sqlite3.Connection) -> IngestSummary`: open the CSV with stdlib `csv.DictReader`
    (streaming, never load the whole file into memory at once), run `map_columns` on
    `reader.fieldnames` once, then per row: build a normalized dict via the column map, run
    `parse_range` on each of the four `_..._range` pseudo-keys present and route the results
    into the matching `*_min_musd`/`*_max_musd` fields (a no-op passthrough — i.e. use the cell
    value directly — if a row already supplies clean numeric `*_min_musd`/`*_max_musd` columns
    instead of a combined range column); skip the row (log a warning, increment
    `rows_skipped`) if neither `firm_name` nor `website` is present; construct a seed
    `FirmRecord` from the row's fields (status left at the pydantic default); call
    `existing = db.get_firm(conn, website)`; call `merged, conflicts =
    merge.merge_firm_record(existing, seed_record)`; if `conflicts`, set `merged.needs_review =
    True` (this explicit override is intentional and documented — merge_firm_record's generic
    lifecycle pass-through does not itself flag conflicts, the caller does, per CONTEXT.md);
    call `db.upsert_firm(conn, merged)`; increment the summary counters accordingly. Return the
    `IngestSummary`. Import `csv`, `re`, `dataclasses`, `pathlib.Path`, `sqlite3` (for type hint
    only) from stdlib; import `merge`, `db`, `models.FirmRecord` from `pescraper`.
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User-supplied local CSV file -> ingest_csv | The file path is supplied directly by the local user of this tool, not attacker-controlled in this local-only threat model; cell contents are untrusted data, not code |
| Parsed CSV values -> db.upsert_firm | Values ultimately reach SQLite through the already-parameterized `upsert_firm`/`get_firm` helpers (02-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
