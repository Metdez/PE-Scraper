## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI argument (url) -> pipeline entry | User-supplied on the local machine; not attacker-controlled in this local-only tool's threat model, but flows into every downstream module (crawl, extract, db) |
| End-to-end data flow: scraped content -> LLM -> pipeline.db | The full chain assembled by this plan; each individual hop is mitigated in its owning plan (02-01 through 02-04) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-10 | Tampering | End-to-end: scraped content through to the persisted firms/extractions rows | high | mitigate | Defense-in-depth already established per-hop in 02-01 (merge/provenance), 02-02 (parameterized SQL), 02-03 (bounded crawl), 02-04 (schema-constrained decoding + sanity clamp) — this plan's only new responsibility is correct wiring order, verified by the Task 1/2 behavior tests plus the human-verify checkpoint |
| T-02-11 | Repudiation | No audit trail for why a firm landed in needs_review | medium | mitigate | Every needs_review determination is logged with a specific reason code (`no_criteria_page`, `low_confidence`, `zero_core_numerics`, `seed_conflict:[...]`) and the CLI summary surfaces status/confidence/needs_review directly — satisfies CONTEXT.md's "flag needs_review with a specific reason code" without adding a 25th column to the locked 24-column schema |
| T-02-12 | Denial of Service | run_firm() hanging indefinitely on a pathological site | low | mitigate | Inherited bound from 02-03 (`page_timeout`, `max_pages`); this plan adds no new unbounded loop |
