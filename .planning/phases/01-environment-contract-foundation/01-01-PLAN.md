<objective>
Stand up the Windows-native Python package skeleton that every later phase inherits: a uv-managed project (`src/pescraper/`) with the `pescraper` console entry point, the Windows runtime hardening baked into package init (asyncio Proactor event-loop policy + UTF-8 I/O), and a runnable typer CLI skeleton with stub subcommands. This delivers Phase 1 success criterion 3 (runnable `pescraper` CLI, Windows entry point proven) and the runtime foundation for the ENVR-01 smoke test.

Purpose: Establish the package layout, dependency set, and Windows runtime conventions before any pipeline code exists. The asyncio Proactor policy and UTF-8 guards are the top two Windows failure modes (Playwright subprocess support + cp1252 mojibake) per PITFALLS.md — baking them into `__init__` means every entry point (CLI, smoke test, later worker) inherits the fix.
Output: An installable `pescraper` package, the `pescraper` CLI with 6 registered commands (run, run-firm, export, status, doctor, init-db), and passing runtime + CLI tests.
</objective>

    Create `src/pescraper/runtime.py` with a single idempotent function `configure_windows_runtime()` that: (a) on win32 sets asyncio.set_event_loop_policy to a WindowsProactorEventLoopPolicy instance only if the active policy is not already Proactor (Playwright needs subprocess support); (b) forces UTF-8 by reconfiguring sys.stdout and sys.stderr to encoding utf-8 with errors=replace and setting os.environ PYTHONUTF8=1 and PYTHONIOENCODING=utf-8 as belt-and-suspenders; (c) returns a diagnostics dict with keys platform, python_version, event_loop_policy, stdout_encoding, pythonutf8. It must be safe to call repeatedly and safe on non-win32 (skip the policy step, still force UTF-8). Do not print anything.

    Create `src/pescraper/__init__.py` that imports `configure_windows_runtime` from runtime and calls it at import time (side effect, so any consumer of the package inherits the hardened runtime), then exposes `__version__ = "0.1.0"`. Keep it minimal.

    After writing files, run `uv sync` to create the venv and install deps, then run `uv run crawl4ai-setup` to install the Playwright Chromium browser into this Windows venv (this is the browser the wave-2 smoke test launches). Do not run crawl4ai-doctor here — that belongs to the smoke-test plan.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PyPI -> uv venv | Third-party packages are installed into the local Windows venv during `uv sync` |
| developer shell -> Windows OS | The runtime mutates process-global event-loop policy and stdio encoding |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-SC | Tampering | uv/pip package installs (crawl4ai, ollama, pydantic, typer, httpx, tenacity, python-dotenv, openpyxl) | high | mitigate | All packages verified legitimate against PyPI in STACK.md (Sources, dated 2026-07-19); pin exact/floor versions in pyproject and commit `uv.lock` so installs are reproducible. No `[ASSUMED]`/`[SUS]`/`[SLOP]` packages introduced — no legitimacy checkpoint required. |
| T-01-01 | Denial of Service | configure_windows_runtime() global policy mutation | low | mitigate | Function is idempotent and win32-guarded; never overrides an already-Proactor policy, avoiding fighting another library that set it |
| T-01-02 | Information disclosure | UTF-8 reconfigure of stdio | low | accept | Local-only console output; no untrusted content flows through stdio in this plan |
</threat_model>
