<objective>
Create the SQLite contract `pipeline.db` — the source of truth the whole pipeline builds against — via an idempotent initializer, plus the pydantic v2 model of the fixed 24-column firm record. Delivers Phase 1 success criterion 2 and requirement DATA-02: the five-table schema (jobs, firms, pages, extractions, cache), the 24-column firms schema, a demonstrable status lifecycle (pending -> in_progress -> complete|needs_review), and a 90-day staleness query for re-queue.

Purpose: The DB schema is the inter-language / inter-phase contract (per ARCHITECTURE.md, "two halves, one contract"). Establishing it now — WAL mode, per-row transactions, crash-safe writes — before any crawl/extract code exists means Phase 2+ writes against a stable, indexed store instead of the whole-file-rewrite JSON approach the reference scraper used. Only the 24-column firms table is fully specified this phase; jobs/pages/extractions/cache are created as skeleton tables (columns later phases extend), because they must EXIST as part of the contract now.
Output: `src/pescraper/models.py` (FirmRecord + FirmStatus), `src/pescraper/db.py` (init_db, connect, status helpers, staleness query), passing db tests, and .gitignore coverage so the Capital IQ-derived store is never committed.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pipeline.db -> git repo | The SQLite store holds Capital IQ-derived firm data that must never enter version control |
| multiple processes -> pipeline.db | Later phases (worker + interactive requests) write concurrently; WAL + busy_timeout prevent corruption |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-03 | Information disclosure | pipeline.db / exports committed to git | high | mitigate | Extend .gitignore to ignore *.db, WAL sidecars, and data/exports; repo stays code-only (closes the reference-scraper gap noted in PITFALLS Security Mistakes) |
| T-01-04 | Tampering | Concurrent writes corrupting the store | medium | mitigate | WAL journal mode + busy_timeout=5000 on every connection; per-call commits keep writes atomic and crash-safe |
| T-01-05 | Tampering | Invalid status transitions corrupting lifecycle state | medium | mitigate | advance_status validates against ALLOWED_TRANSITIONS and raises on disallowed moves rather than silently overwriting |
</threat_model>
